# CI - Test Reports Generated but Not Uploaded as Artifacts

## Justification

The `run_tests.py` script (or a similar testing script integrated into CI) generates test reports (e.g., XML, HTML, or JUnit format). However, these reports are not being uploaded as artifacts in the CI pipeline. Without accessible reports, diagnosing test failures requires manually searching through CI logs, which is inefficient and makes it harder to get a quick overview of test results, trends, or specific failure details. The cost of not fixing this is increased time to diagnose test failures, difficulty in tracking test quality over time, and a less transparent testing process.

## Problem Description

A CI job executes tests using a script like `run_tests.py`. This script outputs test result files to a specific directory within the Docker container or CI runner. However, the CI workflow (e.g., `main.yml`) does not include a step to upload these generated files as build artifacts.

```python
# Hypothetical snippet from run_tests.py
def main():
    # ... logic to run tests ...
    results = runner.run_tests() # Assume this generates a report file
    output_dir = os.getenv("UE_PROJECT_ROOT", "/app") + "/Saved/TestResults"
    if not os.path.exists(output_dir):
        os.makedirs(output_dir)

    report_path = os.path.join(output_dir, "unittest_report.xml")
    with open(report_path, "w") as f:
        f.write(results.to_xml()) # Example: writing an XML report
    print(f"Test report generated at {report_path}")
    # Script exits, but report is only in the container's filesystem
```

```yaml
# Hypothetical snippet from .github/workflows/main.yml
# ... (previous job steps) ...
      - name: Run Unit Tests
        run: python3 Source/RaidGame/Tests/run_tests.py
      # MISSING STEP: No actions/upload-artifact step here for test reports
# ... (subsequent job steps) ...
```

## Agent Task

Modify the CI workflow to upload the test reports generated by `run_tests.py` as build artifacts. This will make the reports easily accessible from the CI job's summary page.

## Sub-Tasks

- [ ] Identify the exact path(s) where `run_tests.py` (or the relevant test script) saves its report files.
    - This might involve inspecting `run_tests.py` or checking CI logs for output messages indicating report locations.
    - Common locations include `Saved/TestResults/` or similar within the project directory.
- [ ] In the CI workflow file (e.g., `.github/workflows/main.yml`), locate the step that executes the tests.
- [ ] Add a new step immediately after the test execution step, using the `actions/upload-artifact` GitHub Action (or equivalent for other CI systems).
- [ ] Configure `actions/upload-artifact`:
    - `name`: A descriptive name for the artifact (e.g., `test-reports`).
    - `path`: The path to the directory or specific file(s) containing the test reports. Wildcards can be used (e.g., `Saved/TestResults/*.xml`).
    - `if`: Optionally, configure it to upload `always()`, `success()`, or `failure()` to ensure reports are uploaded even if tests fail. `always()` is usually best for test reports.
- [ ] Ensure the `path` correctly reflects the location within the CI runner's environment where the files are generated.
- [ ] Trigger a CI run that executes tests (and ideally some failures) to verify that the artifacts are uploaded and accessible.
- [ ] Check the downloaded artifact to ensure it contains the expected report files.

## Files to Examine/Modify

- `.github/workflows/main.yml` (or the primary CI configuration file)
- `Source/RaidGame/Tests/run_tests.py` (or the script responsible for running tests and generating reports, to find the output path)
